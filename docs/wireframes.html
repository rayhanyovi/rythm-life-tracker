Rythm - Full Wireframe Spec

Low-fi structural spec for the `Tasks-first` shell reframe. This artifact is intentionally grayscale, layout-focused, and future-facing where the current backend has not yet caught up to the planned task model.

Core direction: replace the old `Today | Structure | History` shell with a TickTick-inspired workspace: module rail, tasks rail, list column, detail pane.
Why this matters: the previous slim HTML lost too much of the planning context. This version restores broader spec coverage: route mapping, shell behavior, screen-by-screen wireframes, state coverage, and implementation notes.

Compatibility Map

Habit Lists are the user-facing containers for recurring tasks. The current category-backed data model remains an implementation detail until the planned schema work.

Navigation principles

  • The module rail stays intentionally short.
  • Fixed views and list buckets share one tasks rail.
  • `Lists` and `Habit Lists` are context buckets, not peer modules.
  • Identity, sign-out, and scope stay in utility chrome.
  • Mobile collapses rails into a drawer or stacked switcher.
  • The list column remains the dominant working surface.
  • Detail stays contextual instead of generating route churn.
  • `Journal` may appear disabled or placeholder-like.
Desktop shell +----------+----------------+---------------------------------+------------------+ | Modules | Tasks | List / Agenda column | Detail pane | | Tasks | Today | selected view content | selected item | | Journal | Upcoming | rows stay dominant | note / edit | | | Calendar | | | | | Activity Log | | | | | Lists | | | | | Habit Lists | | | +----------+----------------+---------------------------------+------------------+

Working regions

  • Module rail: product scope and future module switching.
  • Tasks rail: fixed task views first, user-owned buckets second.
  • List column: the operational center of gravity.
  • Detail pane: inspect, edit, reschedule, note, and confirm destructive actions.

Mobile shell contract

  • Only one rail is visible at a time.
  • The active tasks rail item becomes the page title.
  • List stays full width and detail opens as a bottom sheet.
  • Module and list switching must be reachable in two taps or fewer.
Mobile rule: collapse both rails into a drawer or stacked switcher. Do not invent a second IA for phones.
Compatibility note `Journal` is allowed to stay conceptual at the routing layer for now. The shell needs to be locked before every surface becomes live code.

Loading and fetch errors

  • Use row-shaped skeletons, not generic card placeholders.
  • Errors name the failed action directly: load Today, load Lists, save note.
  • Recovery action should stay in-context: retry, reopen sheet, or keep editing.
  • Do not dump the user into a blank shell after a fetch failure.

Inline field errors

  • Task title required.
  • List assignment required when the flow expects it.
  • Recurrence controls should explain invalid combinations.
  • Messages appear next to the failing field, not only at the top of the sheet.

Destructive states

  • Delete copy names the exact task or completion being removed.
  • Blocked delete explains the dependency.
  • Destructive confirmation lives in the detail pane or sheet when possible.
  • Successful destructive action should refresh the list in place.

Core UX Principles for This App

The wireframe is only useful if the product-level rules are clear enough to survive implementation details later.

Principle board

Work before chrome
One task language
Context-preserving detail
Lists over galleries
Honest future-facing scope
Empty states route back to value

Rule split

Navigation
Tiny module rail
Mixed tasks rail
Same IA on mobile
Interaction
Quick add near top
Low-drama selection
In-place correction

1. Work before chrome

The first thing users should see is actionable task context, not intro copy, dashboard stats, or setup marketing inside the app shell.

2. One task language

One-off tasks and recurring instances must share one visual grammar so the workspace feels coherent.

3. Context-preserving detail

Inspecting, editing, and correcting should happen in sheets or panes before new routes are introduced.

4. Lists over galleries

Rows and grouped date buckets beat cards for operational task work.

5. Honest future-facing scope

The shell may anticipate richer task models, but labels and copy must not pretend the backend is already there.

6. Empty states route back to value

Every empty state should suggest the next productive step, not just describe absence.

Navigation rules

  • Module rail stays tiny.
  • Tasks rail mixes fixed views and user-owned buckets.
  • Mobile does not get a different information architecture.
  • Journal should not overpower the core `Today` workflow.

Interaction rules

  • Quick add belongs near the top of the active list surface.
  • Row selection should be visually obvious but low-drama.
  • Destructive actions require clarity, not friction theater.
  • Archive correction must preserve the user's place in the feed.

Next Steps and Implementation Notes

This section translates the wireframe into a practical build sequence while keeping the current route compatibility intact.

Implementation sequence board

1. Rework app shell
2. Map current routes into Tasks labels
3. Implement Today as merged surface
4. Move History into Activity Log
5. Keep categories as Lists stand-ins

Open decisions board

One-off task schema
Upcoming horizon depth
Calendar interaction depth (done)
Journal product model

Recommended order

  1. Rework the app shell to `module rail + tasks rail + list column + detail pane`.
  2. Map existing routes into `Tasks` labels before creating new public route structure.
  3. Implement `Today` as the merged recurring-plus-due working surface.
  4. Move `History` behavior into `Tasks / Activity Log`.
  5. Use categories as `Lists` stand-ins until a real broader task model exists.

Open decisions before coding deeper

  • real schema for one-off tasks and due dates
  • how far `Upcoming` should look ahead by default
  • whether Calendar starts month-first or week-first — v1 starts month-first
  • what Journal actually becomes as a product model
Still future-facing: one-off tasks, due dates, Journal data model needs backend definition before full implementation.
Locked now: shell layout, navigation naming, view hierarchy, row grammar, and compatibility mapping.

This document is intentionally low-fi and structurally opinionated. It reduces ambiguity before code changes begin and should stay aligned with DESIGN.md and docs/wireframes.md.
