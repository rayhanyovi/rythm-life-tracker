01

App Structure and Route Mapping

Previous shell direction used exactly 3 authenticated top-level items: `Today`, `Structure`, and `History`.

Target top-level navigation: Today, Structure, History. Categories were no longer allowed to remain a first-class sibling beside daily use and archive review. They became a subview inside Structure. Note editing remained contextual, never promoted into its own primary route.

Top-Level Navigation

  • Today
  • Structure
  • History

Compatibility Map

  • /dashboard -> Today
  • /quests -> Structure / Quests
  • /categories -> Structure / Categories
  • /history -> History

Navigation principles

  • Top-level app nav stayed capped at 3 items.
  • Daily execution outranked management, and management outranked archive review.
  • Utilities like signed-in identity and account scope stayed in shell chrome.
  • Mobile navigation had to expose the same conceptual model as desktop.

Locked structural decisions

  • Today was the conceptual replacement for Dashboard.
  • Structure held Quests and Categories as subviews.
  • History stayed separate because correction was a distinct job.
  • Checklist rows remained the dominant content pattern.

Today

The previous Today surface treated `/dashboard` as the conceptual home for recurring quest execution.

Target first viewport: date, current period context, progress strip, compact filters, then category-grouped checklist rows. Not a KPI wall, not a card gallery, and not a detail panel that steals focus before the user has acted.
Daily-use flow Open app -> See date + progress + visible quests -> Check / uncheck row -> Open note sheet / pane

Structure Surface Overview

`Structure` grouped Quests and Categories into one management concept with segmented subviews.

Structure was the new parent surface for both Quests and Categories. In the live app they still lived on separate routes, but the information architecture converged on a segmented surface with shared shell rules: one toolbar region, one list region, and one contextual editor or detail region.
Structure shell +-----------------------------------------------+ | Structure | | ( Quests ) ( Categories ) | +-----------------------------------------------+ | shared toolbar, shared list, contextual pane | +-----------------------------------------------+

Behavior rules

  • Quests: search, filters, list, create/edit sheet or pane.
  • Categories: ordered list, rename, reorder, delete guard, starter pack.
  • Desktop may add a side pane. Mobile uses sheets and stacked sections.

IA rule

  • Neither subview earned a separate primary nav slot.
  • Categories were support structure, not a daily destination.
  • Structure existed to reduce route hopping for setup work.

Structure / Categories

Categories already supported create, rename, reorder, delete, and seed defaults from the bootstrap endpoint. The API also protected deletion if quests still depended on the category.

History and Correction Flow

`History` was a first-class top-level surface for archive review, note editing, and delete correction.

Key rule: note editing and deletion stayed attached to the selected completion row. Users should never feel like they left the archive to manage data elsewhere.
History flow Set filters -> Review grouped feed -> Select completion -> Edit note / Remove completion / Load more

Desktop

+----------------+-----------------------------------------+------------------+ | Rail | History | Completion pane | | Today | grouped feed + compact filters | note editor | | Structure | | delete record | | History | | | +----------------+-----------------------------------------+------------------+
History emptiness was supposed to route users back to Today or Structure, not trap them in archive mode.

Support Surfaces and Offline Boundary

Support surfaces stayed narrow: they existed to help the user enter, recover, or understand limits. In that product phase, the only support surface that needed a dedicated wireframe outside landing and auth was offline.

Offline

+----------------------------------+ | You're offline | | shell UI may reopen, but writes | | still need network | +----------------------------------+ | [Try sign in again] | | [Retry app entry] | +----------------------------------+

State Coverage

  • Today empty -> open quests, manage categories
  • Quests empty -> add quest, manage categories
  • Categories empty -> seed defaults, create category
  • History empty -> go to Today, open Structure
Why keep this copy: it preserves the prior IA and the richer section coverage that existed before the Tasks-first rewrite.

Empty, Loading, and Error States

The repo already contained several explicit states worth standardizing. Every empty state was supposed to teach the next useful action. Every loading state was supposed to hint at the shape of incoming content. Every destructive or validation error had to state the cause and the next step.

Today empty

  • No quests in the current view.
  • [Open quests]
  • [Manage categories]

Quests empty

  • No quests match the current view.
  • [Add quest]
  • [Manage categories]

Categories empty

  • No categories yet.
  • [Seed defaults]
  • [Create category]

History empty

  • No history in the current view.
  • [Go to Today]
  • [Open Structure]
Skeletons were supposed to mimic list rhythm instead of generic gray blocks. Failure copy was supposed to name the failed action directly.

State Flow Diagrams

These flows described the behavioral backbone of the previous app model. They mattered because quest deletion, completion notes, and category dependencies were semantically strict.

Quest definition lifecycle

  • Create quest -> Active
  • Edit metadata
  • Deactivate
  • Delete with completion cascade

Current-period completion lifecycle

  • Unchecked quest row -> Checked in current period
  • Save note
  • Reset note
  • Uncheck completion

Category lifecycle

  • Create or seed category
  • Rename / reorder
  • Delete blocked if quests depend on it
  • Delete when clear

History correction lifecycle

  • Select completion
  • Edit note or remove completion
  • Archive refreshed in place

Core UX Principles

  • One dominant action per screen
  • Context-preserving detail
  • Lists over galleries
  • Structural honesty
  • Compact filters first
  • Empty states should route back to value

Implementation Notes and Next Steps

This document was still a wireframe artifact, not a coded feature. The next pass was supposed to map these sections to the actual Next.js app with the least route churn necessary.

Recommended implementation order

  1. Relabel Dashboard to Today in shell copy and page hierarchy.
  2. Merge Quests and Categories under a new Structure shell with a segmented switch.
  3. Move note editing into a more mobile-appropriate sheet or drawer on Today.
  4. Trim auth support chrome so the form remained dominant on smaller widths.
  5. Normalize empty, loading, and error states across all core surfaces.

Open decisions worth confirming before coding

  • Would the shell switch from 4 nav items to 3 in one release, or through compatibility labels first?
  • Should desktop Today keep a permanent detail pane, or move to a dismissible drawer for parity with mobile?
  • Would Add Quest remain accessible from Today header once Structure became the main management home?
  • Should the auth split shell survive unchanged, or be toned down for better task focus?
